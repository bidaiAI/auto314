# Autonomous 314 开源发布策略

最后更新：**2026-04-11**

本项目采用 **主开发仓库 + 公开 MIT 发布镜像仓库** 的模式：

- **公开仓库：** [https://github.com/bidaiAI/auto314](https://github.com/bidaiAI/auto314)
- **许可证：** [MIT](../LICENSE)
- **发布方式：** 仅将经过审查的版本手动发布到公开仓库

## 为什么采用这种模式

现在的协议安全前提是：**真正有价值的安全，必须在源码公开后依然成立。**

目标不是用私有代码掩盖生产风险，而是：

1. 日常开发、实验分支、热修复继续留在主开发仓库，
2. 只把经过审查的版本主动发布到公开 MIT 仓库，
3. 把运行时防护而不是“闭源”当作真正的安全边界。

## 哪些内容公开

公开 MIT 仓库用于承载这些对外发布内容：

- 合约源码，
- 前端源码，
- indexer 源码，
- 协议文档，
- 接入文档，
- 经过审查的发布版本快照。

## 哪些内容保持私有

以下内容并不是依赖“闭源”来保证安全，但仍应作为运维资产保持私有：

- secrets 与真实生产环境变量，
- 内部 runbook，
- 事故复盘与应急记录，
- 滥用防护阈值与风控细节，
- 尚未审查的分支与热修复，
- 对外集成不需要知道的基础设施访问细节。

## 核心原则

每个公开版本都应满足以下规则：

1. **默认攻击者可以读到源码。**
2. **生产写接口不能匿名暴露。**
3. **公开配置不能默认把敏感生产内部面直接暴露出去，除非这是有意为之。**
4. **不能公开 secrets、内部专用 env 或应急流程。**
5. **不能把“别人看不到代码”当作 API 安全前提。**

## 公开前最低安全门槛

- 上传 / 触发类写接口必须鉴权，
- 公共 POST 面必须有速率限制与滥用控制，
- 上传内容必须有 TTL / 清理 / 配额机制，
- `/health` 之类的健康接口必须最小化，
- metadata 拉取逻辑必须防 SSRF，
- 文档与公开配置必须做脱敏，
- 相关改动测试通过。

## 建议的本地 remote 结构

保留主开发仓库为 canonical `origin`，再单独增加公开镜像 remote：

```bash
git remote add public git@github.com:bidaiAI/auto314.git
```

## 手动发布流程

建议的发布流程：

1. 在主开发仓库开发和审查，
2. 完成安全修复与脱敏，
3. 检查文档、前端仓库链接、公开默认配置，
4. 将审查通过的状态手动推送到公开仓库，
5. 只公开你希望外部看到的版本。

公开仓库**不是**日常开发过程的实时镜像。
